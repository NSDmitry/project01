# Book Club API

FastAPI + SQLAlchemy + PostgreSQL. Серверные сессии (X-Session-Id), Alembic-only управление схемой (см. readme.md).

## graphify-first: сначала граф, потом код

В репозитории есть построенный граф знаний в `graphify-out/` (узлы, рёбра, сообщества, god-nodes). Перед тем как искать код grep-ом или читать файлы для понимания зависимостей - сначала спроси граф. Это быстрее и показывает связи между модулями, которые не видны при чтении одного файла.

### Когда использовать граф

- **Понять, как устроена фича или подсистема** перед изменением - `query`.
- **Найти, как один модуль связан с другим** (цепочка зависимостей) - `path`.
- **Разобраться, что за сущность и с чем она связана** - `explain`.
- **Найти ключевые абстракции / точки входа** - смотри раздел God Nodes в `graphify-out/GRAPH_REPORT.md`.

### Команды

```
/graphify query "как работает аутентификация"        # BFS обход - широкий контекст
/graphify query "..." --dfs                            # DFS - проследить конкретную цепочку
/graphify path "UserRepository" "DBUser"               # кратчайший путь между двумя сущностями
/graphify explain "AuthService"                        # что это и с чем связано
```

Если граф не содержит ответа - не выдумывай рёбра, переходи к чтению кода.

### Держать граф актуальным

Граф - снимок на момент построения. После изменения кода обнови его, иначе ответы устареют:

```
/graphify . --update      # инкрементально переизвлекает только изменённые файлы
```

Для code-only изменений `--update` работает без LLM (только AST), быстро. Для изменений в `readme.md` или других доках нужна семантическая переэкстракция (тоже через `--update`).

Опционально можно поставить git-хук, чтобы граф перестраивался после каждого коммита:

```
graphify hook install
```

### Артефакты

- `graphify-out/GRAPH_REPORT.md` - god-nodes, сообщества, мосты между подсистемами, аудит рёбер (EXTRACTED / INFERRED).
- `graphify-out/graph.json` - сырой граф (используется командами query/path/explain).
- `graphify-out/graph.html` - интерактивная визуализация.

## Принципы написания кода

Принципы поведения, снижающие типичные ошибки LLM при написании кода. Объединять с инструкциями проекта по необходимости.

**Компромисс:** эти принципы смещают баланс в сторону осторожности, а не скорости. Для тривиальных задач полагайся на здравый смысл.

### 1. Думай перед тем, как писать код

**Не делай допущений. Не скрывай непонимание. Проговаривай компромиссы.**

Перед реализацией:
- Явно проговаривай свои допущения. Если не уверен - спрашивай.
- Если есть несколько трактовок - покажи их, не выбирай молча.
- Если есть более простой подход - скажи об этом. Возражай, когда это обоснованно.
- Если что-то неясно - остановись. Назови, что именно непонятно. Спроси.

### 2. Простота прежде всего

**Минимум кода, который решает задачу. Ничего про запас.**

- Никаких функций сверх того, что просили.
- Никаких абстракций для кода, который используется один раз.
- Никакой "гибкости" или "настраиваемости", которую не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Если написал 200 строк, а можно 50 - перепиши.

Спроси себя: "Сказал бы опытный инженер, что это переусложнено?" Если да - упрощай.

### 3. Точечные изменения

**Трогай только то, что необходимо. Убирай только за собой.**

При правке существующего кода:
- Не "улучшай" соседний код, комментарии или форматирование.
- Не рефактори то, что не сломано.
- Следуй существующему стилю, даже если сделал бы иначе.
- Если заметил несвязанный мёртвый код - упомяни о нём, но не удаляй.

Когда твои изменения порождают "сирот":
- Удаляй импорты/переменные/функции, которые стали неиспользуемыми из-за ТВОИХ изменений.
- Не удаляй ранее существовавший мёртвый код, если об этом не просили.

Проверка: каждая изменённая строка должна напрямую следовать из запроса пользователя.

### 4. Выполнение от цели

**Определи критерии успеха. Повторяй, пока не проверено.**

Превращай задачи в проверяемые цели:
- "Добавить валидацию" → "Написать тесты на некорректный ввод, затем добиться их прохождения"
- "Починить баг" → "Написать тест, воспроизводящий его, затем добиться прохождения"
- "Отрефакторить X" → "Убедиться, что тесты проходят до и после"

Для многошаговых задач сформулируй краткий план:
```
1. [Шаг] → проверка: [что проверяем]
2. [Шаг] → проверка: [что проверяем]
3. [Шаг] → проверка: [что проверяем]
```

Сильные критерии успеха позволяют работать итеративно самостоятельно. Слабые критерии ("просто сделай, чтобы работало") требуют постоянных уточнений.

**Эти принципы работают, если:** в диффах меньше лишних изменений, меньше переписываний из-за переусложнения, а уточняющие вопросы возникают до реализации, а не после ошибок.

## Пунктуация

Никогда не использовать длинное (—) или среднее (–) тире. Везде короткий дефис (-). Правило для кода, документации, коммитов и PR.
